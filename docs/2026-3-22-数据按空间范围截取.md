# 2026-3-22 数据按空间范围截取

## 1. 目标

为单通道 `PHASE` 数据类型模式新增“按空间范围截取”功能。

用户可在 GUI 的 `Phase Demod Parameters` 参数区设置：

- `CropStart`
- `CropEnd`

软件在从采集卡读取单通道 `PHASE` 原始相位数据后，立即只保留指定空间范围的数据，后续的：

- 波形显示
- 时空图显示
- 监控图显示
- 数据存储
- TCP 通信

统一使用截取后的数据。

## 2. 适用范围

本功能只对以下模式生效：

- `upload.channel_num == 1`
- `upload.data_source == PHASE`

以下模式不启用该功能：

- 多通道 `PHASE`
- `Raw / I_Q / arc`

## 3. 参数规则

### 3.1 基本规则

- `CropStart` 为起始空间点
- `CropEnd` 为终止空间点
- 采用区间规则：`[CropStart, CropEnd)`
- 即起点包含，终点不包含

这样做的结果是：

- `CropStart = 100`
- `CropEnd = 230`

则保留原始第 `100 ~ 229` 点，共 `130` 点。

这与用户需求中“原来的第 100 个点重新编号为 1，原来的 230 变为第 130 个点”的数量关系一致。

### 3.2 默认值

- `CropStart = 0`
- `CropEnd = 0`

表示不做空间截取，保留全部相位点。

### 3.3 越界处理

- 若 `CropEnd > 总点数`，则自动截断到总点数
- 若 `CropStart >= 总点数`，启动时参数校验报错
- 若 `CropEnd > 0` 且 `CropEnd <= CropStart`，启动时参数校验报错

### 3.4 点数基准

空间截取不是对原始 `point_num_per_scan` 直接做，而是对 `PHASE` 输出点数做。

即先经过：

- `merge_point_num`

得到：

- `point_num_after_merge = point_num_per_scan // merge_point_num`

再在该点数基础上执行软件空间截取。

## 4. 数据流设计

### 4.1 处理顺序

单通道 `PHASE` 的处理顺序为：

1. 采集线程从采集卡读取完整 `PHASE` 数据块
2. 采集线程将一维数据视为 `frame_num x point_num_after_merge`
3. 采集线程按列切片执行一次空间截取
4. 将截取后的连续内存数据发给 GUI
5. GUI、存储、通信统一使用截取后的数据

### 4.2 关键原则

- 截取只做一次
- 截取不放在主线程
- 不在绘图、存储、通信线程中重复裁剪

这避免了：

- GUI 再切一次
- 存储前再切一次
- TCP 发包前再切一次

## 6. 性能分析

### 6.1 对软件处理压力的影响

该功能可以明显降低以下环节的压力：

- Qt 主线程收到的数据量
- 波形绘制点数
- 时空图矩阵宽度
- 存储写盘字节数
- TCP 通信发送字节数

以“采集 7000 多点，只关心末尾 200 多点”为例，后续软件处理数据量会显著下降。

### 6.2 对底层采集带宽的影响

该功能不能减少以下部分：

- 采集卡到 PC 的底层 DMA 读取量
- DLL 接口一次读取的原始完整相位块大小

原因是当前底层接口仍然是先读取完整 `PHASE` 数据块，再在软件采集线程中裁剪。

因此本功能优化的是：

- 软件处理压力

不是：

- 设备上传带宽本身

### 6.3 为什么将截取放在采集线程

这是当前架构下最合适的位置，因为它同时满足：

- 比主线程更早
- 比存储线程更早
- 比 TCP 线程更早
- 只裁一次

这样可以最大化减少后续所有链路的负担，并避免多个线程重复裁剪导致的不一致。

## 7. 线程设计说明

当前线程职责如下：

- 采集线程：读取完整相位块并执行一次空间截取
- GUI 主线程：接收截取后数据并做显示调度
- 存储线程：异步写入截取后数据
- TCP 线程：异步发送截取后数据

本次改动没有改变原有线程分工，只是在采集线程内增加了一个轻量级矩阵列切片步骤。

因此：

- 不会把额外负担转移到 GUI 主线程
- 不会影响异步存储线程模型
- 不会影响异步 TCP 线程模型

## 8. 使用示例

### 8.1 不截取

- `CropStart = 0`
- `CropEnd = 0`

结果：

- 显示全部单通道 `PHASE` 数据
- 保存全部单通道 `PHASE` 数据
- 通信发送全部单通道 `PHASE` 数据

### 8.2 截取局部范围

假设：

- `point_num_per_scan = 20480`
- `merge_point_num = 25`

则：

- `point_num_after_merge = 20480 // 25 = 819`

若设置：

- `CropStart = 100`
- `CropEnd = 230`

则实际输出：

- 保留第 `100 ~ 229` 点
- 每帧点数变为 `130`

后续：

- 绘图按 130 点解释
- 文件名按 130pt 记录
- 存储数据每帧只写 130 点
- TCP 包按 130 点上下文构建

## 9. 当前结论

该功能方案在当前工程架构下是可行的，且实现方式合理：

- 截取只做一次
- 截取位置足够靠前
- 不放在主线程
- 后续显示、存储、通信统一使用截取后数据

对于“大量采集、只关心局部空间范围”的实际使用场景，可以有效降低软件运行压力并改善卡顿风险。
