## 3. 存储链路总览

当前主链路如下：

1. 设备数据由采集线程从 DLL/API 读取。
2. 采集线程通过 Qt signal 将数据发到 GUI 主线程。
3. GUI 主线程在 `_on_phase_data()` 或 `_on_raw_data()` 中调用 `data_saver.save_frame(data)`。
4. `FrameBasedFileSaver` 将数据转成字节流后放入队列。
5. 独立后台存储线程从队列取出数据并顺序写入磁盘。

## 5. 二进制进制、字节序与兼容性

### 5.1 进制

- 文件为二进制，不存在十进制/十六进制文本表示。
- 读取时应使用 NumPy 二进制读取，例如 `np.fromfile(path, dtype=np.int32)`。

### 5.2 大小端

当前实现没有显式写入大小端标记，使用的是 NumPy/平台原生 `int32` 内存字节序：

- 代码调用：`data.tobytes()`
- 当前运行平台为 Windows/x86，实际落盘为小端序
- 但文件格式本身不自描述大小端

工程建议：

- 在当前 Windows/x86 环境下，按小端 `'<i4'` 读取最稳妥
- 若未来跨平台迁移到非常规大端平台，应在离线脚本中显式指定 dtype

### 5.3 无头文件约束

由于文件无头部，以下信息不在文件内存储：

- `scan_rate`
- `frame_num`
- `points_per_frame`
- `channel_num`
- `data_source`
- `merge_point_num`
- `diff_order`
- `detrend_bw`
- `polarization_diversity`
- 采集起止绝对时间

这些信息只能通过以下来源恢复：

- GUI 截图/日志
- 单独的外部记录文件

## 6. 数据排列方式

### 6.1 单通道 PHASE

单个保存块的典型 shape 为：

- `(frame_num, points_per_frame)` 在业务理解上成立
- 实际落盘前可能表现为一维或二维数组，但 `tobytes()` 按 NumPy C-order 顺序展开

对于当前常见配置：

- `scan_rate = 2000 Hz`
- `frame_num = 2000`
- `points_per_frame = 819`
- `channel_num = 1`

若 `Frames/File = 10`，理论完整文件 shape 为：

- `20000 x 819`

理论元素数：

- `10 * 2000 * 819 = 16,380,000`

理论字节数：

- `16,380,000 * 4 = 65,520,000 bytes`

### 6.2 多通道数据

多通道时，采集线程会先 reshape 成：

- `(frame_num * points_per_frame, channel_num)`

然后按 C-order 展开写盘，因此磁盘顺序为交错通道布局。

## 7. 数据是否经过预处理

### 7.1 软件保存前处理

软件在保存前做的处理非常少，主要只有：

- 非 `int32` 时强制 `astype(np.int32)`
- `tobytes()` 转换为字节流
- 放入存储队列

软件侧不会在保存前做：

- 文本化
- 浮点化
- 归一化
- GUI `rad` 换算
- 二次滤波
- PSD 计算

## 8. 线程模型与职责

### 8.1 线程分工

当前存储相关线程分工如下：

- 采集线程：负责从设备读取 DMA/相位数据
- GUI 主线程：负责响应 Qt signal、触发 `save_frame()`
- 存储线程：负责真正写盘和分文件

### 8.2 存储线程模型

当前实现使用单个后台线程串行写盘：

- 队列类型：`queue.Queue`
- 数据入队：`put_nowait()`
- 写盘线程：`threading.Thread(..., daemon=True)`
- 分文件请求：通过 `_split_marker` 控制消息进入同一队列

### 9.3 已修复问题：分文件边界丢帧

历史问题：

- 旧实现中，调用线程在边界时直接关旧文件并开新文件
- 后台线程可能仍在写旧文件
- 表现为 `write to closed file`
- 结果是每个完整文件在边界丢 1 个保存块

修复方式：

- 分文件改为通过 `_split_marker` 排入同一保存队列
- 由保存线程顺序执行切文件
- 避免“旧文件先关闭、边界块后写入”的竞争条件

## 10. 文件名命名规则

### 10.1 当前规则

当前主文件名格式：

- `{seq}-eDAS-{rate}Hz-{points}pt-{timestamp}.{ms}.bin`

实际示例：

- `000041-eDAS-2000Hz-0819pt-20260320T165509.633.bin`

字段说明：

- `seq`：6 位序号，零填充
- `eDAS`：固定标识
- `rate`：扫描率，单位 Hz
- `points`：每帧点数
- `timestamp`：调用 `datetime.now()` 时的墙钟时间
- `ms`：毫秒

对应代码：

- [src/data_saver.py](/D:/OneDrive%20-%20EVER/00_KY/DAS-%E5%99%A8%E4%BB%B6%E6%B5%8B%E8%AF%95%E3%80%81%E4%B8%8A%E4%BD%8D%E6%9C%BA%E5%BC%80%E5%8F%91/BX/PCIe-7821/pcie7821_gui/src/data_saver.py#L365)

### 10.2 文件名时间的含义

文件名中的时间不是“首样本时间”也不是“末样本时间”，而是：

- 创建该文件名时的墙钟时间
- 对首文件，是开始保存时刻
- 对后续文件，是分文件请求被保存线程处理时刻

因此：

- 连续文件名时间间隔不要求严格等于 10.000 s
- 几毫秒级抖动属于正常线程调度与系统时钟误差
- 不能仅凭文件名时间间隔判断是否丢数据

- `convert_phase_to_radians()`：按 GUI 同一公式换算为弧度
- `read_single_channel_phase_bin()`：直接返回弧度相位

## 13. 文件完整性校验建议

建议按以下优先级检查：

1. 检查日志中是否出现 `write to closed file`
2. 检查 `Dropped` 计数是否非零
3. 检查文件字节数是否等于理论值
4. 检查 `np.fromfile(...).reshape(...)` 是否与理论 shape 一致
5. 检查连续文件是否存在 0 byte 或异常小文件

对于单通道 PHASE、`scan_rate=2000`、`frame_num=2000`、`points_per_frame=819`、`Frames/File=10`：

- 理论完整文件为 `20000 x 819`
- 理论字节数为 `65,520,000`

## 14. 当前实现的工程约束

当前存储实现有以下工程约束：

- 文件无头部，不自描述参数
- 文件名只包含有限元信息
- 默认以平台原生小端 `int32` 落盘
- 无块级时间戳
- 无内建 CRC 或校验和
- 无 sidecar metadata 文件
- Raw 路径当前也会被转为 `int32`
- TimedFileSaver 为遗留实现，主工程当前使用 FrameBasedFileSaver

## 15. 建议的后续演进

若后续对可追溯性和长期维护要求更高，建议增加：

- 每文件 metadata sidecar，例如 `.json`
- 文件头，写入版本号、dtype、endianness、shape、channel_num、scan_rate
- 块级时间戳或帧号
- 校验和或 CRC
- 明确区分 `raw int16` 与 `phase int32` 的落盘格式
- 启动/停止采集时自动生成参数快照

## 16. 结论

当前 eDAS 主存储链路的本质是：

- 将 GUI 接收到的相位块数据按 `int32` 原始二进制顺序写盘
- 由单独后台线程异步保存
- 以固定块数分文件
- GUI 的弧度显示换算不影响存盘
- 文件名时间是墙钟时间，不是数据时间戳
- 分文件边界丢帧问题已修复，但队列满导致的丢块风险仍然存在

因此，软件工程上应把当前 `.bin` 文件视为：

- 无头、平台相关、参数外置的工程二进制文件

后续所有离线解析、归档、跨平台共享与质量追踪，应以此约束为前提。
